# 2026-07-03 회의 — 작업/회의 기록 체계 잡기

> 참여: 치즈랩 대표, Claude
> 주제: 클로드코드 작업을 어떻게 기록·관리하고, 나중에 관리자 페이지에서 열람할 것인가

---

## 배경 (왜 이 회의를 했나)

Gemini와의 긴 대화(히타치 엑셀 지옥 → SpaceX식 데이터 구조화)를 정리하다가,
"클로드코드 작업 내역을 자동으로 남겨서 나중에 확인하고 싶다"는 욕구가 나옴.
이게 무엇인지(티켓? 회의록?) 머릿속에서 정리가 안 된 상태로 회의 시작.

---

## 오늘 정한 것 (결론)

1. **대표가 찾던 개념의 정체 = 티켓.**
   Gemini 원문에 이미 나와 있던 개념이었음(이슈 트래커/티켓). 새로 만든 게 아님.
   - 티켓 = "무슨 일을 할 거냐"의 상태 (todo→doing→done)
   - 회의록 = "무슨 얘기를 나눴나"의 기록
   - 둘은 다른 축이고, 서로 재료가 됨 (회의 → 거기서 티켓이 태어남)

2. **구현은 외부 SaaS(Jira 등)가 아니라 repo 안 마크다운으로.**
   Gemini는 Jira/Notion을 권했으나, 대표 실제 환경(클로드코드+repo+Firebase)에선
   repo 안 `.md` 파일 + git 커밋 히스토리로 더 가볍게 같은 걸 달성.
   → "관리하는 걸 또 관리하는" 히타치 마트료시카를 피하는 유일한 지점.

3. **역할 분담 확정.**
   - 전략·회의·정리 = 여기(클로드 앱)
   - 코드·파일저장·상태갱신 = 클로드코드 (CLAUDE.md 규칙으로 자동)
   - 대표가 실제로 하는 것 = "정리해줘" / "커밋해줘" 두 마디 수준

4. **자동화의 정체 = CLAUDE.md에 규칙 한 번 심기.**
   "작업 끝나면 상태 done + 로그 한 줄 추가" 규칙을 심어두면 그때부터 자동.
   안 심으면 수동 싱크가 부활 → 그 순간이 곧 히타치 늪.

5. **관리자 페이지 열람 = 나중에 창문만 붙이면 됨.**
   worklog/티켓이 텍스트(마크다운)로 있으므로 SSOT-창문 구조 그대로 적용.
   원본이 먼저 쌓여야 화면이 채워짐. 화면부터 설계하지 않는다.

6. **가장 중요 — 지금은 과설계하지 않는다 (SpaceX 원칙).**
   티켓 템플릿/회의록 스킬/폴더구조/화면 레이아웃을 미리 확정하려던 것 자체가
   "데이터도 없는데 양식부터 만드는" 히타치 함정이었음. 대표가 스스로 감지해 멈춤.
   → **지금 실제로 깔 것은 `worklog.md` 한 개.**
   → 티켓·회의록·화면은 진짜 불편이 올 때 얹는다(접어두는 것이지 버리는 것 아님).

---

## 아직 안 정한 것 (다음에, 필요해지면)

- 티켓의 정확한 칸 구성 (id/status/우선순위 외 무엇을 더)
- 회의록과 티켓을 한 파일에 묶을지 폴더로 나눌지
- 관리자 페이지에서 작업 로그를 어떤 화면(표/카드/필터)으로 보여줄지
- 회의록을 매번 남길지, 티켓 '왜' 칸으로 대체할지

→ 이 항목들은 **worklog 2~3주 운영 후** 실제 불편을 근거로 결정.

---

## 확인된 구조 (오늘 OK 난 5가지)

| 항목 | 상태 | 방식 |
|------|------|------|
| 티켓 관리 | OK | repo 안 마크다운, 클로드코드가 발행 |
| 회의록 관리 | OK | 여기서 회의 → "정리해줘" 한 줄로 파일화 |
| 서로 재료가 되냐 | OK | 회의록에서 티켓이 태어나고 번호로 연결 |
| 자동화 | OK | CLAUDE.md 규칙 → 상태·로그 자동, git이 박제 |
| 관리자 페이지 열람 | OK | 원본이 텍스트라 나중에 창문만 붙이면 됨 |
