# 2025-12-21 회의 — Firestore DB 검토

> 참여: 치즈 (단독, ChatGPT 상담)
> 주제: 시트를 DB처럼 쓰는 현 구조에서 구글 무료 DB 옵션 검토 → Firestore로 갈지
> ※ 이관 노트. Notion 스크린샷 기반 대화를 회의록 형식으로 재구성. (현 관리자 시스템의 DB가 실제로 Firestore이므로 이 검토의 실현체.)

---

## 배경 (왜 이 회의를 했나)

지금은 구글 스프레드시트를 DB처럼 쓰고 있는데, 데이터가 쌓이면 목록 조회가 느려지고 조건 검색·페이지네이션·프론트 직접 접근이 불편하다. 구글에서 쓸 수 있는 무료 DB 옵션을 검토했다.

---

## 오늘 정한 것 (결론)

1. **구글 공식 무료 DB = Firebase Cloud Firestore(사실상 유일한 정답).**
   NoSQL 문서형 DB, REST API/SDK 제공, 프론트에서 직접 접근 가능, 실시간 동기화 + 인덱스 기반 조회. 무료 티어(Spark): 읽기 5만/일·쓰기 2만/일·저장 1GB — 개인 프로젝트·사내툴엔 충분.

2. **시트보다 Firestore가 나은 지점.**
   시트는 "행 전체 스캔"이라 목록이 느리고 조건 검색·페이지네이션을 직접 구현해야 하며 프론트 직접 접근·권한이 약함. Firestore는 조건 조회 빠름·페이지네이션 기본·프론트 접근·구조 변경 유연·권한 강력.

3. **점진적 하이브리드 구조 채택.**
   마스터/원장/백업 = Google Sheet(사람이 보기 좋음) · 목록/조회/대시보드 = Firestore(앱이 빠르게 읽는 DB) · 자동화/승인/메일 = Apps Script · 무거운 처리 = Render. "시트=원장, Firestore=앱이 읽는 DB"로 역할 분리.

4. **대안 DB는 제외.**
   Realtime Database(Firestore 구버전 느낌, 복잡한 필터엔 약함) / Cloud SQL·BigQuery(유료·실시간 앱 DB 부적합) → 지금 시작이면 Firestore가 정답.

---

## 아직 안 정한 것 (다음에, 필요해지면)

- **Firestore 유료(Blaze) 요금 수준**: 원문 말미 질문("유료면 어느 정도?")은 미해결.
- **NoSQL 스키마 설계**: SQL 아님(JOIN 없음), 문서형 사고방식으로 스키마 재설계 필요.
- **마이그레이션 범위**: 어떤 데이터를 시트에 남기고 어떤 걸 Firestore로 옮길지.

→ Firestore 스키마 설계와 시트↔Firestore 역할 분담 확정이 다음 단계.
