# 2026-02-15 회의 — AI 블로그 글쓰기 에디터 (도입~2.1)

> 참여: 치즈랩 대표 (기획: AI Assistant)
> 주제: 블로그 글쓰기에 AI를 붙여 작성 시간을 줄이고, 유사도 문제 없는 독창 콘텐츠를 만든다

> ※ 사후 정리 노트. 세 개의 연속 기획서(도입 2/14 · 2.0 2/15 · 2.1 2/15, `report_archive` DRIVE 원문)를 하나의 여정으로 묶어 정리했다.
> 회의록·티켓 체계 도입(7월) 이전 기록이라 티켓 번호는 없다.

---

## 배경 (왜 이 회의를 했나)

블로그 글 한 편을 쓰는 데 시간이 너무 많이 들고, 반복적인 내용이 계속 나왔다.
게다가 기존 글과의 유사도 문제로 독창적인 콘텐츠를 뽑기 어렵고, 작성자의 개성 있는 톤앤매너를 반영하는 것도 비효율적이었다.
→ 에디터 안에 AI 글쓰기 기능을 내재화해, 작성 시간을 줄이면서 톤앤매너·가독성이 최적화된 고유한 글을 만들자는 것.
이 회의는 도입(1차) → 2.0(고도화) → 2.1(재추진)로 이어진 세 번의 기획을 한 흐름으로 묶은 기록이다.

---

## 오늘 정한 것 (결론)

1. **(도입, 2/14) 에디터에 AI 글쓰기를 내재화한다.**
   툴바에 AI 진입점 + 상세 입력 모달을 두고, 서버단에서 Gemini API를 호출한다.
   핵심은 프롬프트 엔지니어링 — 톤앤매너·가독성을 프롬프트로 제어한다.

2. **(2.0, 2/15) 1차 한계를 보완한다.**
   1차에서 "글 품질이 프롬프트에 크게 의존한다"는 게 확인됨.
   → 프롬프트 가이드라인을 고도화하고, 워크플로우를 **문단 단위 생성 → 전체 글 생성** 방향으로 개선 검토. 약 5~6주 개발 예상.

3. **(2.1, 2/15) 2.0 목표 미달 위험 → MVP로 재추진한다.**
   2.0이 목표(품질·UX)를 다 달성하지 못할 위험이 상존 → 핵심 기능에 집중한 '최소 기능 제품(MVP)' 전략으로 재추진.
   명확한 글쓰기 품질 기준을 세우고, 애자일(개발-테스트-반복)로 빠르게 돌린다.

---

## 아직 안 정한 것 (다음에, 필요해지면)

- 글쓰기 품질의 **정량 기준** (무엇을 "좋은 글"로 볼 것인가 — 2.1의 핵심 미결)
- 문단 단위 생성 vs 전체 글 생성 — 최종 선택
- Gemini API 사용 비용 상한 / 호출 최적화

→ 2.1 MVP를 돌리며 사용자 피드백과 실제 생성 결과를 근거로 확정.
