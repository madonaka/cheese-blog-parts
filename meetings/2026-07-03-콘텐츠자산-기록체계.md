# 2026-07-03 회의 — 콘텐츠·DB 작업 기록 체계

> 참여: 치즈랩 대표, Claude
> 주제: 코드 커밋이 아닌 콘텐츠 작성·DB 입력 작업을 어떻게 기록·관리하고, AI가 재활용하게 할 것인가

---

## 배경 (왜 이 회의를 했나)

현재 log_manage(작업 관리)에 쌓이는 기록은 전부 **코드 수정 = git 커밋** 기준이다.
하지만 프로젝트의 핵심 작업 중 하나는 **콘텐츠를 써서 Firestore에 입력하는 일**이고,
이건 커밋 과정이 아니라 로그에 안 남는다. "이런 것도 기록할 수 있나"에서 출발해,
실제 집필 동선(미국사·러시아편 시리즈)을 뜯어보며 "무엇을 남길 가치가 있나"까지 파고들었다.

---

## 오늘 정한 것 (결론)

1. **남길 가치가 있는 건 "입력 결과물"이 아니라 "시리즈 설계도"다.**
   코드는 최종 상태만 보면 뭘 했는지 안 보여 커밋 로그가 필요했지만,
   콘텐츠는 결과물(에피소드)이 Firestore에 그대로 남아 열람된다 → **결과물이 곧 로그**.
   따라서 "몇 편 입력했다"는 별도 로그는 저가치·중복. 정작 DB에 안 남는 건
   그 편을 만든 **설계·구상**(척추·괴리 구조·편별 연표·톤 규칙 등)이다.

2. **집필 워크플로는 클로드 웹 그대로 유지한다.**
   한 번 시작하면 3~4시간짜리 순수 본문 집필이라, 코딩 도구(클로드 코드)에 안 맞는다.
   → 집필은 웹 스킬(usa/russia/ussr/roc-series-writer 등)에서 그대로. 무대를 옮기지 않는다.

3. **설계도의 원본(SSOT)은 Firestore의 "시리즈"에 붙인다 (repo md 아님).**
   집필이 웹에 남으므로, 설계도를 repo md로 빼면 그 에피소드들(Firestore)과 또 갈라진다.
   → 설계도는 자기 에피소드가 사는 곳(Firestore `learn_series`)에 붙는 게 맞다.
   (repo md는 클로드 코드로 집필할 때 유효했던 안 — 제약이 바뀌어 폐기)

4. **결합 방식 = "예치(deposit)". 시리즈 편집 화면에 '설계도 칸'을 만들고 붙여넣는다.**
   실시간 동기화가 아니라, 웹에서 완성한 설계도를 시리즈당 한 번 붙여넣는 체크포인트.
   대표가 이미 에피소드 본문을 learn_manage에 붙여넣는 습관과 동일. 설계도는
   시리즈당 1개·거의 고정이라 부담도 작다.

5. **별도 "입력 로그"는 만들지 않는다.**
   시리즈+설계도+에피소드+진행상태가 관리자 한곳에 모이면, 제작 이력이 자동 성립.
   앞부분 논의의 "저장 길목 자동로그"보다, 결과물 자체가 로그가 되는 이 구조가 더 단순.

---

## 확인된 구조 (이번에 파악된 것)

**설계도 ↔ 관리자 데이터 모델이 이미 1:1로 포개진다** (새 시스템이 아니라 '설계층'만 얹으면 됨):

| 시리즈 마스터 설계도의 부분 | 관리자에 이미 있는 그릇 |
|------|------|
| 정체성·척추·괴리 구조 (설계도 1~3장) | 시리즈 (`learn_series`) |
| 편별 상세 설계 (연표·인물·괴리) | 각 에피소드 (`learn_sequences`) |
| 진행 상태 (본문 미착수 / 집필중 / 발행) | 에피소드별 제작 상태 = 제작 로그 |

이렇게 하면 대표 니즈 3가지가 동시에 풀린다:
- **한곳 관리** — 시리즈 열면 [설계도 + 에피소드 + 상태]가 한 화면
- **로그 대체** — 진행 상태가 곧 제작 로그, 결과물이 곧 기록
- **재활용** — 설계도가 Firestore에 있어 AI가 읽고 ①다음 편 집필 ②"제작기" 메타콘텐츠 ③타 시리즈 일관성 참조

참고: 콘텐츠 자산 전체 목록(학습청크·인물사건DB·이슈·상품·챗봇지식 등)은 별첨 격.
이번 결론은 그중 **시리즈형 학습 콘텐츠**에 우선 적용. 비시리즈(상품·챗봇)는 이 틀 밖.

---

## 아직 안 정한 것 (다음에)

- **'설계도 칸'의 구현 형태** — 단순 md 텍스트칸 하나 vs 진행 상태만 살짝 구조화(편별 todo/집필중/발행)
- **재예치 규칙** — 설계도가 진화할 때(설계도 13장) 언제 다시 붙여넣나, 버전 흔적을 남길지
- **방법 vs 설계 분리 적용** — 웹 스킬에 남길 공통 규칙(톤·명명·작업절차)과 시리즈별 설계도의 경계선
- **비시리즈 콘텐츠**(상품 상세·챗봇 지식)는 이 구조 밖 — 별도로 다룰지

→ '설계도 칸'을 실제 시리즈 편집 화면에 얹을 때 결정.
