# 2026-07-03 회의 — 콘텐츠·DB 작업 기록 체계

> 참여: 치즈랩 대표, Claude
> 주제: 코드 커밋이 아닌 콘텐츠 작성·DB 입력 작업을 어떻게 기록하고, 나중에 AI가 분석·활용할 것인가

---

## 배경 (왜 이 회의를 했나)

현재 log_manage(작업 관리)에 쌓이는 기록은 전부 **코드 수정 = git 커밋** 기준이다.
worklog의 한 줄은 "한 커밋 = 한 repo"를 전제로, 뷰가 그 커밋 해시로 GitHub API를 때려
변경 파일을 펼쳐 보여준다.

그런데 프로젝트의 핵심 작업 중 하나는 **콘텐츠를 써서 DB(Firestore)에 입력하는 일**이다.
이건 커밋 과정이 아니라 로그에 안 남는다. "이런 작업도 기록할 수 있나, 어떻게?"에서 회의 시작.

---

## 오늘 정한 것 (결론)

1. **콘텐츠/DB 작업도 기록 대상에 포함한다.**
   기술적으로는 이미 가능함 — 뷰의 파서가 커밋 해시 없는 줄도 정상 렌더링한다(해시 자리에 "커밋 없음"만 뜸).
   즉 "기록 가능하냐"의 답은 이미 Yes. 진짜 문제는 *어떤 형태로, 어디에, 누가* 쌓느냐다.

2. **기록은 "저장 길목"에 자동으로 얹는 방식으로 간다 (수동 몰아쓰기 아님).**
   코드 작업이 자동으로 남았던 이유 = **커밋이라는 확정된 길목**에 로그를 얹었기 때문.
   콘텐츠도 똑같은 원리 — **DB에 넣으려면 저장 버튼을 무조건 한 번은 누른다**(대표 지적).
   그 확정된 길목(Firestore 저장)에 기록을 얹으면 "까먹음" 변수가 원천 제거된다.
   → 나중에 몰아서 불러주는 수동 방식(유실 위험)보다 견고하다고 판단.

3. **이 기록의 최종 목적 = 단순 열람이 아니라 "AI가 과거 자산을 분석해 새 콘텐츠를 생성"하는 재료다.**
   대표가 md를 고집했던 이유가 이것 — Firestore는 grep도 안 되고 AI가 통으로 읽기 나쁘다.
   그래서 기록은 사람용 로그가 아니라 **AI가 읽기 좋은 형태(md 지향)**여야 한다는 요구가 확정됨.
   이 지점에서 논의가 "저장 로그" → "콘텐츠 자산 대장(corpus)"으로 한 단계 올라감.

---

## 아직 안 정한 것 (다음에 — 대표가 더 탐구하기로 함)

- **로그(메타데이터)로 충분한가, 코퍼스(본문 포함)여야 하는가.**
  "제목·URL 목록"만으론 AI가 새 글을 생성 못 함 → 본문 자체가 필요할 수 있음. 이게 갈림길의 핵심.
- **저장소 방식.** Firestore를 원본으로 두고(저장 길목에서 자동 축적) → md는 그 투영(export/스냅샷)으로
  주기 생성하는 안이 제안됨. "저장 자동성 + md 분석성 + 본문 포함"을 다 잡는 절충안이나 **미확정**.
- **관리 대상 자산의 범위.** 아래 8개 전부를 하나의 코퍼스로 볼지, 역사 콘텐츠 계열만
  (학습 청크 + 인물·사건 + 이슈)으로 좁힐지.
- **모든 저장을 기록할지, 신규만 할지** (오타 수정까지 쌓이면 노이즈).
- **어느 페이지부터 심을지** (저장 지점이 여러 개 — 한 번에 다 말고 하나부터).
- **공개 URL 없는 콘텐츠**(챗봇 지식 등)의 앵커를 docId로 대체할지.

→ 대표가 "이건 나중에 좀 탐구해봐야겠다"고 판단, 급하게 결론내지 않고 보류.

## 확인된 구조 (이번에 파악된 것 — 콘텐츠 자산 전체 목록)

거래/운영 데이터(주문·회원·QnA·환불·결재)는 콘텐츠 자산이 아니라 제외. 코퍼스 후보는 다음.

| 자산 | 컬렉션 / 관리 페이지 | 성격 |
|------|------|------|
| 학습 콘텐츠 본문(=청크) | `learn_sequences` / `_full` / `learn_series` / `learn_index` | 역사배우기 본문 |
| 학습 인물·사건 사전 | `learn_character_manage` (=인물사건DB) | 사전형 |
| 독서 인물 기록 | `character_manage` | 사전형 |
| 독서 사건 기록 | `book_event_manage` | 사전형 |
| 독서/학습 노트 | `study_notes` | 원재료 |
| 이슈·사건 DB | `issue_db` (E-Force, 월별 총정리) | 시의성 |
| 상품 상세 | `products.detailed_content` | 판매 콘텐츠 |
| 챗봇 지식 청크(RAG) | 임베딩 컬렉션 | 파생물 |

- new_dashboard가 이미 "콘텐츠 자산 현황"(인물DB·독서DB·이슈/학습 시리즈)을 세고 있음 — 대표 구상이 코드에 반쯤 존재.
