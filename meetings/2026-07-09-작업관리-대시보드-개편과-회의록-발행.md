# 2026-07-09 회의 — 작업 관리 대시보드(티켓 탭) 개편과 회의록 발행 방식 표준화

> 참여: 치즈랩 대표, Claude
> 주제: log_manage 작업 관리 화면을 보기 편하게 개편하고, 회의록을 쉽게 발행하는 방법을 정리한다.

---

## 배경 (왜 이 회의를 했나)

작업 관리 대시보드(`log_manage.html`)가 데이터가 쌓일수록 보기 힘들어졌다. 탭마다 전체를 한 번에 렌더해 페이징이 없었고, 티켓은 할 일/진행 중/완료 3섹션을 세로로 쌓아 완료가 늘수록 스캔이 어려웠다. 여기에 더해, 다른 챗(웹 Claude)에서 만든 회의록을 관리자 페이지에 올리는 방법이 불명확했고, 예전에 "회의록 발행해줘"라고 하면 엉뚱한 작업을 하던 문제도 있어 발행 방식을 정리할 필요가 있었다.

---

## 오늘 정한 것 (결론)

1. **티켓 탭은 '상태 필터칩 + 게시판식 페이징'으로 간다.**
   칸반 3열·접이식 섹션 대신, 상단에 전체·할 일·진행 중·완료 칩(개수 배지)을 두고 한 상태씩 페이징한다. 모바일 친화적이고 페이징과 궁합이 좋다는 이유로 이 안을 택했다. 4개 탭(작업 로그·티켓·회의록·제작 현황) 전부에 페이징을 넣는다.

2. **진행 중(doing) 티켓 강조 배너를 둔다.**
   검색·필터와 무관하게 티켓 탭 상단에 고정하고, 클릭하면 진행 중 필터로 이동한다. "지금 뭐 하던 중이었지"를 항상 보이게 하려는 목적. 세션을 이어받는 활성 티켓(SSOT)과 같은 개념이다.

3. **티켓 카드는 기본 접힘, 클릭하면 로그·회의록이 펼쳐진다.**
   헤더(번호·상태·제목)와 요약 카운트(로그 N·회의 M)만 기본 노출하고, 눌러야 상세가 열린다. 목록을 짧게 유지해 스캔을 쉽게 한다.

4. **티켓 ↔ 작업 로그 ↔ 회의록을 양방향으로 연동한다.**
   티켓 카드의 로그 줄 → 작업 로그 탭 해당 항목, 티켓의 '관련 회의록'(그 티켓 번호를 언급한 회의록) → 회의록 탭, 작업 로그 탭의 티켓 배지 → 티켓 카드. 이동 시 대상 탭의 해당 페이지로 점프하고 펼침+하이라이트한다.

5. **회의록 '업로드 UI'(회의록 탭에서 파일을 올려 `meetings/`에 커밋)는 만들지 않는다.**
   페이지가 GitHub Pages 정적 호스팅이고 repo가 공개라, 쓰기 권한 GitHub 토큰을 클라이언트에 두면 노출된다. 백엔드 프록시로는 가능하지만 기존 Claude Code/git 작성 흐름과 중복되고 보안 민감 경로가 늘어, 필요 대비 이득이 적다고 판단했다.

6. **회의록 발행은 `/회의록` 슬래시 명령으로 표준화한다.**
   한글 명령으로 만들고, 회의 원문을 붙여넣거나 `.md` 경로를 주는 방식으로 입력한다. 원문이 없으면 지어내지 않고 되묻는다. 절차는 규격화(meeting-notes-writer 형식)→`meetings/` 저장→worklog 기록→`git status` 확인→커밋→푸시(자동)까지 한 번에. 예전에 "발행해줘"가 엉뚱하게 동작한 원인(약한 트리거 + 원문 부재 시 임의 추론)을 이 명령이 차단한다.

## 아직 안 정한 것 (다음에, 필요해지면)

- 더 짧은 명령 별칭(`/mn` 등) 추가 여부 — 지금은 `/회의록` 하나만 둔다.
- 회의록을 UI로 자주·직접 올릴 일이 생기면, 백엔드(GAS/Cloud Function) 발행 엔드포인트 검토 — 현재는 보류.

→ 실제 사용 빈도를 보고 필요해지면 결정한다.

## 확인된 구조 (이번에 OK 난 것)

| 항목 | 상태 | 방식 |
|------|------|------|
| 탭별 페이징 | OK | 4개 탭 게시판식 페이징 |
| 티켓 뷰 | OK | 상태 필터칩 + 페이징 + 카드 접기 |
| 진행 중 표시 | OK | doing 배너(상단 고정, 클릭=진행 중 필터) |
| 티켓↔로그↔회의록 | OK | 탭 이동 + 대상 펼침 + 하이라이트, 양방향 |
| 회의록 발행 | OK | `/회의록` 명령(정규화→저장→worklog→커밋→푸시) |
| 회의록 업로드 UI | 보류 | 정적 호스팅·공개 repo 보안 + 중복으로 미채택 |
