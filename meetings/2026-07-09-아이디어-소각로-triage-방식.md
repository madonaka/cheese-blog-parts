# 2026-07-09 회의 — 아이디어 함 "소각로(triage)" 방식 확정

> 참여: 치즈랩 대표, Claude
> 주제: 활용 못 하고 쌓이기만 하는 아이디어 함을, 새 repo 시스템과 어떻게 이을 것인가

---

## 배경 (왜 이 회의를 했나)

문서 아카이브 정리(기획서·보고서 → 회의록 백필 후 삭제)에 이어, 아이디어 함(todo_manage.html)도
타임라인 로그와 겹치는지 검토하다 시작됨. 대표의 문제의식: **"아이디어 함 기능 자체는 필요한데(가볍게 적는 느낌),
지금은 활용을 못 하고 일일이 수동으로 체크해야 한다. 이걸 새로 만든 시스템에 녹여낼 수 있나?"**
검토 중 더 큰 그림("치즈랩 아카이브 센터")과, 이 시스템의 뼈대가 SpaceX식이라는 대표 인식이 나와,
SpaceX가 실제로 아이디어를 어떻게 기록·관리하는지 조사까지 이어짐.

---

## 오늘 정한 것 (결론)

1. **아이디어 함은 시트에 그대로 둔다 (repo 마크다운 이관 안 함).**
   전부 markdown으로 밀어넣으면 "지금 해야 할 일"과 "가볍게 적어둔 생각"이 섞인다.
   캡처층(시트=빠른·폰·read/write)과 결정층(repo=git 박제, 소수 정예)을 **일부러 분리**한다.

2. **다리 = 주기적 "대화 triage" (자동 sync·대량 이관 아님).**
   관리자 UI는 git에 못 쓰므로 자동 다리는 무겁다. 대신 사람이 판단하는 주기적 정리를 다리로 삼는다.
   Claude가 미실현 아이디어를 훑어 → "이건 프로젝트 티켓감인가?" 회의 → 발행 결정된 것만
   repo 티켓(T-NNN)으로 올리고 → 그 아이디어는 시트에서 실현 처리.

3. **근거 = SpaceX 원칙. 이건 "저장고"가 아니라 "소각로"다.**
   조사 결과 SpaceX의 비결은 화려한 아이디어 기록 도구가 아니라, 오히려 **기록을 최소화하고 빠르게 결정으로 태우는 문화**:
   - **삭제가 먼저 (머스크 알고리즘 2단계 + 10% 룰)**: 존재하면 안 되는 걸 최적화·자동화하는 게 최대 실수. → 아이디어 대부분은 티켓이 될 게 아니라 방치·삭제돼야 정상. 소수만 승격.
   - **문서 최소화 ("requirements" 금지, "Design Criteria")**: 두꺼운 문서보다 빠른 정보 흐름. → 대표가 우려한 "섞임"을 SpaceX도 피함.
   - **가장 가까운 사람이 즉시 결정**: 직급 높은 사람 아니라 문제에 가까운 사람이 즉석에서. → 우리 triage 회의 = 대표+Claude가 그 자리에서 판단.
   - **결정·교훈만 보존 (무비난 post-mortem)**: 나머지는 안 남김. → repo=결정층, 시트=캡처층.

4. **더 큰 그림 = 치즈랩 아카이브 센터 (단, 점진적).**
   흩어진 시트 기록(아이디어·할일·목표·문서)을 repo 새 시스템 축으로 통합해 "아카이브 센터"로 격상하는 방향.
   단 174 할일·52 아이디어를 한 번에 밀어넣지 않는다(= 히타치 마트료시카 회피). 오늘의 아이디어 triage가 그 첫 조각.
   참고: `log_manage.html`("작업 관리")은 이미 repo 마크다운(worklog·tickets·meetings)을 읽는 **창문**이다. 옛 시트 타임라인 로그(422건)는 지금 유령 상태.
