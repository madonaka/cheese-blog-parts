# 2025-12-14 회의 — 지출건(월중 누적) 위에 지출결의(월 1회 묶음) 기능 추가 구상

> 참여: 치즈 (단독, ChatGPT 상담)
> 주제: v0.4의 예산요청↔지출건 구조 위에, 월말에 지출건들을 묶는 "지출결의" 레이어 추가
> ※ 이관 노트. Notion 구상 대화(ChatGPT)를 회의록 형식으로 압축. v0.4 실제 함수명은 원문도 추측 표기.

---

## 배경 (왜 이 회의를 했나)

현재 v0.4는 예산요청(BUDGET_REQUESTS) ↔ 지출건(EXPENSES)까지만 있고, 지출건을 "최종 단위"로 승인해버린다. 실무의 "월말에 지출건들을 한 번 더 묶어 결의하는 단위"를 넣으려면, 지출건 위에 **지출결의(월 1회, 묶음)** 레이어를 추가해야 한다.

---

## 오늘 정한 것 (결론)

1. **시트 구조 = A안(단순).**
   EXPENSES에 `report_id`(+조회용 `report_period_ym`) 컬럼 추가 + `EXPENSE_REPORTS`(월말 묶음 마스터: report_id·period_ym·total_amount·item_count·status·결재선 등) 신규 1장. (B안 = EXPENSE_REPORT_ITEMS 매핑 시트는 "한 지출을 여러 결의로 쪼개기"가 필요할 때만.)

2. **업무 흐름 = 월중 ready → 월말 결의 생성 → 승인.**
   월중: 지출건을 `ready`까지 등록(예산 차감 안 함). 월말: "결의서 만들기"로 `status=ready` & `report_id` 없는 그 달 지출을 자동 수집 → report_id 채우고 EXPENSE_REPORTS 1행 생성(합계·건수) → 제출 → 재무/관리자가 결의서 1건만 승인.

3. **예산 차감 시점을 '결의서 승인 시점'으로 고정.**
   결의서가 approved 되는 순간에만 포함 지출을 일괄 approved 처리하고 BUDGET_SUMMARY의 spent_amount를 증가시킨다. "월말 확정"과 숫자가 딱 맞음. (기존 "지출 1건 승인 시 차감"에서 트리거 이동.)

4. **핵심 함수·검증 룰.**
   `createExpenseReportFromApproved_(periodYm)` + `submit/approve/rejectExpenseReport()` (budget_request 승인 로직 패턴 재사용). 검증: 결의 포함 라인의 request_id는 예산요청 approved여야 / 라인합 = 지출 총액 / budget_id별 지출이 remaining 초과 시 승인 불가·경고 / 승인 후 period_ym 락.

---

## 아직 안 정한 것 (다음에, 필요해지면)

- **예산 차감 시점 최종 선택**: 옵션1(지출건 승인 시, 실시간 통제·반려 시 되돌림 필요) vs 옵션2(결의 승인 시, 월말 확정과 일치) — 문서는 옵션2 권장하나 운영 철학 따라 갈림.
- **A안 vs B안**: 단순 report_id(A) vs 매핑 시트(B) 최종 확정.
- **v0.4 실제 함수명/모드명**: 원문에 v0.4 전체 스크립트가 없어 정확한 이름은 추측.

→ A안 기준으로 EXPENSE_REPORTS 헤더 + Apps Script 함수(생성/제출/승인/반려 + Web API mode)를 v0.4 스타일로 통째로 이어붙이는 게 다음 단계.

## 확인된 구조 (이번에 OK 난 것)

| 레이어 | 단위 | 승인 |
|--------|------|------|
| 예산요청 (BUDGET_REQUESTS) | 사전결재 | 기존 그대로 |
| 지출건 (EXPENSES, +report_id) | 영수증 단위, 월중 ready | 결의에 묶임 |
| 지출결의 (EXPENSE_REPORTS, 신규) | 월 1회 묶음 | 승인 시 예산 차감·지출 일괄 approved |

화면: 지출 등록(월중, ready 저장) 유지 + `expense-report.html`(미결의 지출 목록·결의 생성·receipt_bundle_url·제출/승인/반려) + INBOX 뷰(submitted만).
