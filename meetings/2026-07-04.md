# 2026-07-04 회의 — 티켓 목록이 안 뜬 원인과 manifest 대안 검토

> 참여: 치즈랩 대표, Claude
> 주제: 관리자 작업관리 페이지에서 티켓 목록이 갑자기 안 뜬 이유와, GitHub API 의존을 없앨지

---

## 배경 (왜 이 회의를 했나)

작업관리 페이지 티켓 탭이 갑자기 "티켓 목록을 불러오지 못했습니다"로 비었음.
원인을 확인해 보니 티켓/회의록 목록을 **GitHub 비인증 API(시간당 60회)**로 가져오는데
그 한도가 소진돼(403) 막힌 것. 대표가 "이걸 아예 API 없이 할 수는 없냐"고 물어
manifest 방식이 무엇이고 더 효율적인지 논의함.

---

## 오늘 정한 것 (결론)

1. **원인 = GitHub API 한도 소진, 티켓이 사라진 게 아님.**
   목록 조회만 API를 쓰고(파일 내용은 Pages에서 직접 읽음), 그 API가 60회/시간에 걸림.
   약 한 시간 뒤 자동 리셋되면 그대로 복구됨.

2. **직접 원인은 새로고침 버튼의 과도한 캐시 우회 → 이미 수정함.**
   새로고침이 API URL까지 cache-bust 해 매번 실제 호출 → 한도를 빨리 소진시킴.
   bust는 Pages 파일(worklog·티켓·회의록 .md)에만 남기고 **API 호출은 제외**하도록 고침.
   이걸로 정상 사용에선 한도에 잘 안 걸림.

3. **manifest 방식은 "지금은 적용 안 함", 일단 둔다.**
   API를 완전히 없애는 대안은 있으나(아래), bust 수정으로 급한 불은 껐으므로
   지금 바꾸지 않고 두기로 함. 앞으로도 가끔 한도에 걸리면 그때 적용.

---

## 아직 안 정한 것 (다음에, 필요해지면)

- manifest 방식을 실제로 도입할지 (또 한도에 걸리는 일이 반복되면 도입)
- 도입 시 어느 형태로 할지:
  - **파일명 manifest** (`tickets/index.json`에 파일 이름만) — 상태 변경 때는 안 건드려도 됨(권장 후보)
  - **전체데이터 manifest** (id·title·status까지 담기) — 요청 1회로 끝나지만 상태 바꿀 때마다 갱신 필요

→ 한도에 반복해서 걸리는 게 확인되면, 파일명 manifest를 기본안으로 도입 검토.

## 확인된 구조 (이번에 OK 난 것)

| 항목 | 상태 | 방식 |
|------|------|------|
| 캐시 우회 범위 | OK | bust는 Pages 파일 fetch에만, GitHub API 호출은 제외 |
| 403 안내 | OK | 한도 초과 시 "잠시 후 다시 시도" 메시지 표시 |
| 커밋 상세 조회 | 유지 | 여전히 API 사용(단 줄 클릭할 때만) — 목록 호출이 빠져 한도 여유 커짐 |
